## 一、项目简介

本项目是一套**基于扩散模型的虚拟试衣系统**,用户在浏览器中上传或拍摄人物正面照,
选择(或上传)一件服装图片,系统通过部署在远程 GPU 服务器上的 OOTDiffusion 模型生成
人物穿上该服装的合成效果图。

### 1.1 应用场景

- **校园展示与教学演示**:支持桌面浏览器与 65 寸大屏触控屏两种交互形态
- **服装电商虚拟试穿**:用户上传自己照片即可预览不同款式上身效果
- **技术设计**:涵盖 Web 前端、后端、AI 推理对接的端到端项目实践

### 1.2 整体定位

| 维度 | 说明 |
|------|------|
| 部署形态 | 三层分离(浏览器 / 中转后端 / GPU 推理服务) |
| 数据规模 | 轻量级 JSON 数据库,适合数十至数百条衣服与模特记录 |
| 用户角色 | 普通用户(试穿) + 管理员(资源库管理) |
| 安全级别 | 适合内网展示与课程交付,生产环境需另行加固 |
| 跨平台 | 后端跨平台,前端任何现代浏览器,触控屏支持全屏交互 |

## 五、后端架构详解

后端位于 `virtual-tryon/backend/`,基于 FastAPI,**仅依赖文件系统作存储**(无 SQL/NoSQL 数据库),适合轻量场景。

### 5.1 应用入口 `main.py`

主要职责:
1. 初始化 FastAPI 应用与 CORS 中间件
2. 自动创建 `static/`、`static/clothes/`、`static/models/` 目录
3. 挂载 `/static` 静态目录到磁盘
4. 注册管理后台路由(`admin.router`)
5. 提供 3 个核心接口:
   - `GET /` — 服务状态
   - `GET /health` — 健康检查(同时探测两个远程推理服务)
   - `GET /api/clothes` — 公开衣服列表(支持 `?category=` 过滤)
   - `GET /api/models` — 公开模特列表
   - `POST /api/tryon` — **核心试穿接口**,转发到远程推理服务

#### 双模型路由策略

```python
REMOTE_TRYON_URL_HD = "http://127.0.0.1:9000/tryon"   # upperbody → HD
REMOTE_TRYON_URL_DC = "http://127.0.0.1:9001/tryon"   # lowerbody / dress → DC

def pick_remote_url(category: str) -> str:
    if category == "upperbody":
        return REMOTE_TRYON_URL_HD
    return REMOTE_TRYON_URL_DC
```

**为什么要分两个**:HD 模型用 VITON-HD 数据集训练,**仅支持上半身**但效果最好;DC 模型用 Dress Code 数据集训练,支持三类但上半身略逊。前端选择不同 category 时自动路由到对应的服务,**用户无感知**。

#### 错误处理

- 502 — 无法连接远程服务(提示用户检查 SSH 隧道)
- 504 — 远程服务超时(默认 5 分钟,可改 `REMOTE_TIMEOUT`)
- 400 — 文件类型 / category 校验失败
- 500 — 保存结果图片失败

### 5.2 数据存储层 `storage.py`

**核心设计**:
- 单文件 JSON 数据库 `db.json`,所有读写都包在 `threading.RLock` 内
- 写入用"先写临时文件再 `os.replace`"的**原子替换**模式,避免并发损坏
- **首次启动自动创建** `db.json`(`DEFAULT_DB` 模板),无需任何初始化命令
- 老版本 db 自动迁移:启动时检测到没有 `models` 字段会自动补齐

#### 暴露的 API

```python
# ── 衣服 CRUD ─────────────────────────────────
list_clothes(category: Optional[str] = None) -> List[dict]
get_cloth(cloth_id: str) -> Optional[dict]
gen_id() -> str                              # 生成 c_xxx 前缀的 id
add_cloth(*, id, name, category, filename, url) -> dict
update_cloth(cloth_id, **fields) -> Optional[dict]
reorder_clothes(ordered_ids: List[str]) -> int
delete_cloth(cloth_id: str) -> Optional[dict]

# ── 模特 CRUD ─────────────────────────────────
list_models() -> List[dict]
get_model(model_id: str) -> Optional[dict]
gen_model_id() -> str                        # 生成 m_xxx 前缀的 id
add_model(*, id, name, filename, url) -> dict
update_model(model_id, **fields) -> Optional[dict]
reorder_models(ordered_ids: List[str]) -> int
delete_model(model_id: str) -> Optional[dict]

# ── 常量 ──────────────────────────────────────
CATEGORIES        = {"upperbody", "lowerbody", "dress"}
DEFAULT_CATEGORY  = "upperbody"
CATEGORY_TO_INT   = {"upperbody": 0, "lowerbody": 1, "dress": 2}
```

#### 数据完整性

- 所有写入操作均在 `with _lock:` 块内执行(防止两个并发请求同时修改 db.json)
- 临时文件命名 `db.json.tmp`,写完后通过 `os.replace()` 原子替换
- 单条记录字段约束:`category` 不合法时回退到 `DEFAULT_CATEGORY`
- `order` 字段为整数,新增项 `order = min(existing_orders) - 1`(放到最前)
- `created_at` 为 UTC ISO 8601 字符串

### 5.3 鉴权与后台 `admin.py`

#### 认证机制

```
登录: POST /api/admin/login { password }
   → 返回 { token, ttl }, token 存入内存 _TOKENS dict

后续请求: 必须在 Header 携带 Authorization: Bearer <token>

Token 配置:
   ADMIN_PASSWORD     默认 "admin123"     可通过环境变量覆盖
   ADMIN_TOKEN_TTL    默认 604800 秒(7天) 可通过环境变量覆盖
```

**特点**:
- Token 仅存内存,**进程重启后所有人需重新登录**(对小型管理后台够用)
- 每次访问自动 GC 过期 token(`_gc_tokens()`)
- 用 `secrets.token_urlsafe(32)` 生成,避免可预测
- 用 `secrets.compare_digest()` 校验密码,避免时序攻击

#### 路由结构

```
POST   /api/admin/login              # 登录,返回 token
POST   /api/admin/logout             # 登出,移除 token
GET    /api/admin/check              # 检查 token 有效性(用于前端启动时校验)

# 衣服(需 Bearer)
GET    /api/admin/clothes            # 列表
POST   /api/admin/clothes            # 批量上传(multipart, 多文件 + category + 可选 name)
PATCH  /api/admin/clothes/{id}       # 改名 / 改 category / 改 order
PUT    /api/admin/clothes/order      # 批量重排(ids 数组)
DELETE /api/admin/clothes/{id}       # 删除(同步移除磁盘文件)

# 模特(需 Bearer,与衣服对称,无 category 字段)
GET    /api/admin/models
POST   /api/admin/models             # 批量上传(无 category)
PATCH  /api/admin/models/{id}        # 仅支持改名 / 改 order
PUT    /api/admin/models/order
DELETE /api/admin/models/{id}
```

#### 上传逻辑

- 校验文件 `content_type` ∈ {jpg, png, webp, jpg}
- 用 PIL 读取并转 RGB(避免透明通道问题)
- **衣服**缩放到长边 ≤ 800px,JPEG 质量 88
- **模特**缩放到长边 ≤ 1024px,JPEG 质量 90(更高保真度,利于人脸/姿态识别)
- 用 `gen_id()` / `gen_model_id()` 生成新 UUID 前缀文件名
- 写入对应目录 + 调用 `storage.add_*()` 写入 db.json

### 5.4 远程推理对接

后端 `main.py` 收到 `/api/tryon` 请求后的完整流程:

```
1. 接收前端 multipart 表单:person + cloth + category
2. 校验文件类型与 category 值
3. 转换 category 字符串 → 整数(0/1/2,对齐远程接口)
4. 根据 category 选择 HD 或 DC 远程 URL
5. 用 requests.post() 转发原始字节流,timeout=300s
6. 接收远程返回的 base64 PNG
7. 解码 base64 → PIL Image → 保存为 result_<uuid8>.jpg 到 static/
8. 返回给前端 { code: 0, data: { image_url: "/static/result_xxx.jpg" } }
```

错误捕获分级:`ConnectionError` → 502, `Timeout` → 504, 其他 → 500/502。

## 六、前端架构详解

前端位于 `virtual-tryon/frontend/`,基于 Vue 3 Composition API + Vite,**完全 SPA**。

### 6.1 应用入口与路由

#### `main.js`

```js
createApp(App).use(router).mount('#app')
```

#### `router/index.js` — 4 条路由

```
/                  → HomeView          桌面试衣主页
/kiosk             → KioskView         触控大屏全屏模式
/admin/login       → AdminLoginView    管理员登录
/admin             → AdminView         后台管理(requiresAuth: true)
*                  → redirect /        404 兜底
```

#### `CameraCapture.vue` 核心算法

```js
// OOTDiffusion 标准输入尺寸
const OUTPUT_W = 768
const OUTPUT_H = 1024
const TARGET_RATIO = 0.75

// 拍照时按 3:4 中心裁切视频帧
const videoRatio = vw / vh
if (videoRatio > TARGET_RATIO) {
  // 视频更宽 → 裁掉左右
  sh = vh;  sw = vh * TARGET_RATIO
  sx = (vw - sw) / 2;  sy = 0
} else {
  // 视频更窄 → 裁掉上下
  sw = vw;  sh = vw / TARGET_RATIO
  sx = 0;   sy = (vh - sh) / 2
}
ctx.drawImage(video, sx, sy, sw, sh, 0, 0, 768, 1024)
canvas.toBlob(...)  // 输出标准 768×1024 JPEG
```

**关键点**:
- 镜像选项仅影响预览,**最终输出不镜像**(避免姿态翻转干扰推理)
- 拍照时画面与预览的 `object-cover` 完全一致(所见即所得)
- 输出尺寸刚好是 OOTDiffusion 的训练输入分辨率,避免额外 resize

## 七、AI 推理服务

### 7.1 OOTDiffusion 简介

[OOTDiffusion](https://github.com/levihsu/OOTDiffusion) 是一个基于扩散模型的虚拟试穿方案,2024 年由小红书与中山大学联合发表。核心思路:

1. **OpenPose** 提取人体 18 个关键点
2. **HumanParsing** 生成人体部位语义分割(头/上身/下身/手臂...)
3. **OOTDiffusion(U-Net + VAE)** 在保留人体姿态与身份的前提下,把目标服装"贴"到对应部位

提供两个权重:

| 权重 | 训练数据集 | 支持类别 | 效果 |
|------|----------|---------|------|
| HD | VITON-HD | 仅 upperbody | 上半身最佳质量 |
| DC | Dress Code | upperbody / lowerbody / dress | 三类通用,上半身略逊 HD |

### 7.2 远程服务 `api_server.py`

部署在 GPU 服务器的 `OOTDiffusion/run/api_server.py`。**懒加载模型**(首次请求时加载,后续复用)。

#### 启动参数

```
--gpu_id     GPU 编号(默认 0)
--port       监听端口
--host       监听地址(默认 127.0.0.1,仅本机)
--model_type hd / dc
--n_steps    扩散步数(默认 20)
--n_samples  每次生成张数(默认 1)
--scale      引导强度(默认 2.0)
```

#### 推荐部署:同时跑两个实例

```bash
# 终端 1 — HD 模型
python api_server.py --gpu_id 0 --port 9000 --model_type hd --n_samples 4 --scale 2.0

# 终端 2 — DC 模型
python api_server.py --gpu_id 0 --port 9001 --model_type dc --n_samples 4 --scale 2.0
```

或使用项目自带的 `start_models.sh`(下面会讲)。

#### 接口契约

```
GET  /              服务状态(model_type 等元信息)
GET  /health        健康检查 + 模型加载状态
POST /tryon         核心接口
     form-data:
       person:    UploadFile  人像图
       cloth:     UploadFile  衣服图
       category:  int  0=upperbody, 1=lowerbody, 2=dress
       n_steps:   int  可选,覆盖启动参数
       n_samples: int  可选
       scale:     float 可选
       seed:      int  可选,-1=随机
     返回:
       { "code": 0, "msg": "success",
         "data": { "image_base64": "<PNG base64>" } }
```

**约束**:`model_type='hd'` 仅接受 `category=0`,否则 400 错误。

### 7.3 双模型路由策略(本地后端)

本地后端 `main.py` 自动根据 category 选择正确的远程服务:

```python
def pick_remote_url(category: str) -> str:
    if category == "upperbody":
        return REMOTE_TRYON_URL_HD   # → :9000
    return REMOTE_TRYON_URL_DC       # → :9001
```

**前端无感知**:用户只需选"上半身/下半身/连衣裙",底层路由自动完成。

### 8.1 Schema(v2)

```jsonc
{
  "version": 2,                   // schema 版本,启动时检测自动迁移
  "clothes": [
    {
      "id":         "c_8be2d84943",         // 随机 UUID,c_ 前缀
      "name":       "白色短袖",              // 显示名称(最长 60 字符)
      "category":   "upperbody",            // upperbody / lowerbody / dress
      "filename":   "c_8be2d84943.jpg",     // 物理文件名
      "url":        "/static/clothes/c_8be2d84943.jpg",
      "order":      0,                       // 排序权重(越小越靠前)
      "created_at": "2026-04-23T06:26:30+00:00"
    }
  ],
  "models": [
    {
      "id":         "m_aca5c57b9e",         // m_ 前缀,区分模特
      "name":       "示例模特 1",
      "filename":   "m_aca5c57b9e.jpg",
      "url":        "/static/models/m_aca5c57b9e.jpg",
      "order":      0,
      "created_at": "2026-05-22T17:19:59+00:00"
    }
  ]
}
```

### 8.2 自动迁移

`storage._read()` 启动时检测:
- 若文件不存在 → 用 `DEFAULT_DB` 创建
- 若缺 `models` 字段 → 自动补齐为 `[]`
- 若缺 `clothes` 字段 → 自动补齐为 `[]`

**意味着新机器克隆下来什么都不用做,直接启动即可。**

### 8.3 并发安全

```python
_lock = threading.RLock()

def _write(data):
    tmp = DB_PATH.with_suffix(".json.tmp")
    with tmp.open("w", encoding="utf-8") as f:
        json.dump(data, f, ensure_ascii=False, indent=2)
    os.replace(tmp, DB_PATH)  # 原子替换,POSIX 保证
```

`os.replace()` 在 Windows 与 Linux 上都是原子操作,即便程序在写入过程中被强制中断,
`db.json` 也不会处于半写入状态。

### 8.4 关于 SQLite / MySQL 迁移

如果未来数据量增大(超过 1 万条记录或多用户并发写入),建议:

1. 把 `storage.py` 的接口签名保持不变
2. 内部从 JSON 切换到 SQLite(`sqlalchemy` 或 `peewee`)
3. 上层 `admin.py` 与 `main.py` 零修改

这是当前设计的"渐进式可演化"特点。

## 十三、部署建议

### 13.1 当前部署形态

```
┌──────────────────┐         ┌──────────────────┐
│  Windows 服务器   │  SSH    │  Linux GPU 服务器 │
│  - 前端 :5173    │ ──隧道── │  - HD 模型 :9000  │
│  - 后端 :8000    │         │  - DC 模型 :9001  │
└──────────────────┘         └──────────────────┘
        ▲
        │ 局域网访问
   ┌────┴────┐
   │ 用户浏览器 │
   └─────────┘
```

### 13.2 生产部署改造点

如要部署到公网生产环境,**最少**还需:

1. **改默认密码**
   ```bash
   export ADMIN_PASSWORD="<强密码>"
   ```

2. **缩小 CORS 范围**(`main.py`)
   ```python
   allow_origins=["https://yourdomain.com"]
   ```

3. **前端 Vite 改为 Nginx 反代**
   - `npm run build` 生成 `dist/`
   - 用 Nginx 托管,把 `/api/*` 反代到 `http://127.0.0.1:8000`

4. **后端用守护进程**
   - Linux:`systemd` 或 `pm2`
   - Windows:NSSM(把 uvicorn 注册为 Windows 服务)

5. **HTTPS 证书**(摄像头 `getUserMedia` 在非 https / 非 localhost 下不可用!)
   - Let's Encrypt 免费证书 + Nginx

6. **GPU 服务器开启 firewall**,只允许中转后端 IP 访问 9000/9001 端口

### 13.3 静态资源大小估算

| 项目 | 大小 |
|------|------|
| 前端 `dist/`(production build) | ~200KB(gzip) |
| 单张衣服 / 模特图 | ~100-300 KB(已自动压缩) |
| 单次推理结果 | ~80-150 KB |
| OOTDiffusion 模型权重 | ~5-8 GB |

## 十四、常见问题

### Q1. 前端能打开但提示"无法连接远程推理服务"

依次检查:
1. `curl http://localhost:8000/health` 看 `remote_hd / remote_dc` 状态
2. 是否打开了 SSH 隧道?(`netstat -ano | findstr 9000`)
3. GPU 服务器上两个 `api_server.py` 是否真的在跑?(`ps aux | grep api_server`)
4. GPU 服务器上 `tail -f logs/hd.log` 是否已经显示 `Uvicorn running on ...`

### Q2. 上传衣服后前台 Dock 看不到

- 浏览器强刷(Ctrl+Shift+R)
- 检查后台是否选择了正确的 category
- 后台 Dock 里"全部"标签应该能看到所有衣服(无论 category)

### Q3. 摄像头无法打开

- **必须使用 HTTPS 或 localhost**(浏览器安全限制)
- 系统设置中检查浏览器是否被允许访问摄像头
- 是否有其他程序占用摄像头(如 Zoom / 飞书)
- 错误信息会提示具体原因(`CameraCapture.vue` 中的 `handleMediaError`)

### Q4. 管理员登录后立刻被踢出

- Token 有效期 7 天,**但后端进程重启会清空所有内存中的 token**
- 重启后端后所有管理员需要重新登录

### Q5. db.json 被意外删除

- 后端启动时会自动用 `DEFAULT_DB` 创建空数据库
- 但 `static/clothes/` 和 `static/models/` 里的物理图片仍在,需要管理员手动重新上传录入

### Q6. 不同机器 git pull 时报冲突

- 本项目的 `.gitignore` 已经把 `db.json`、`static/clothes/`、`static/models/` 都
  排除在 git 之外,**只要不手动 force add 这些文件,就不会冲突**
- 详见 `CHANGELOG_2026-05-23.md` 第十章"多机协作与 Git 建议"

### Q7. Windows 上 `start_dev.bat` 双击没反应

- 用文本编辑器打开看一下,确保没有被 CRLF/LF 混淆
- 用 PowerShell 或 cmd 直接 `start_dev.bat` 运行看错误输出
- 检查 conda 是否已经初始化 cmd shell(`conda init cmd.exe`)

### Q8. Linux 上 `start_models.sh` 报 "找不到 conda.sh"

- 脚本默认尝试 6 个常见路径(`~/anaconda3`, `/opt/miniconda3` 等)
- 若你的 conda 装在其他位置,运行 `conda info --base` 看实际路径
- 手动编辑 `start_models.sh` 的 `CONDA_SH` 候选列表,加入你的路径

### 15.2 技术演进的几个关键决策

1. **为什么用 JSON 而非 SQLite?**
   项目当前规模小(几百条数据),JSON 直观可读、零依赖。`storage.py` 已封装为抽象接口,
   未来切换 SQLite 只需替换实现,上层零改动。

2. **为什么前后端分开 5173 / 8000 两个端口?**
   开发期 Vite 热重载需要独立 dev server。生产部署时建议用 Nginx 反代统一到 80/443。

3. **为什么用本地中转后端转发推理请求?**
   - 隔离推理服务的鉴权与限流策略
   - 推理服务可以只监听 127.0.0.1,减少攻击面
   - 后端可以缓存、记录、限流、补字段
   - 未来切换到其他试穿模型(IDM-VTON / CatVTON / Leffa)时,前端零改动

4. **为什么模特图缩放到 1024px 而衣服只到 800px?**
   人像需要保留人脸 / 姿态等关键细节,OOTDiffusion 输入是 768×1024,
   1024px 长边可以避免再次放大;衣服图细节相对不敏感,800px 已足够 Dock 缩略图展示。

5. **为什么 Token 只存内存而不用 JWT?**
   - 项目规模小,管理员只有 1-2 个
   - 内存存储可以做"远程吊销"(JWT 难以做到)
   - 重启后所有人重新登录是合理的安全代价

### 15.3 后续可选优化方向

详见 `virtual-tryon/PROJECT_FUTURE_ROADMAP.md`,简要清单:

1. **接入更新的试穿模型**(IDM-VTON / CatVTON / Leffa)
2. **试穿历史 / 收藏夹**(扩展 db.json 或迁移 SQLite)
3. **配置文件外置**(把 SSH 目标、端口、模型路径抽到 `.env`)
4. **从 `examples/` 一键批量导入**到运行时库
5. **多用户系统**(目前是单管理员,可扩展为多角色)
6. **结果图水印 / 防盗用追溯**
7. **指标采集**(Prometheus 暴露推理时长、错误率)
